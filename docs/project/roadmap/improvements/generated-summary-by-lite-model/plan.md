# Plan: Generated summary by lite model

## Problem

Two separate short-form representations of an identity layer existed, each with a limitation:

- `firstLine` (in `adapters/web/pages/map.tsx`) takes the first non-empty line and surfaces it on the Cognitive Map card. For `self/soul` it shows the markdown header `# Alma`, which is unhelpful.
- `extractPersonaDescriptor` (in `server/personas.ts`) takes the first non-header line, truncated to 120 chars, for reception's persona-routing classifier. For Template B personas (tecnica, dba, dona-de-casa) the first non-header line is the same independence declaration ("Esta persona opera em registro..."), making the personas indistinguishable to the classifier.

See [Spike §9.2](../../spikes/spike-2026-04-18-identity-lab.md#92-generated-summary-by-lite-model-for-cards-and-routing) for the original analysis.

## Solution

A single `summary` field on each identity layer, generated by the existing lite model (`title` role, Gemini Flash Lite) on Save, used by both the card display and the reception descriptor.

The generation is fire-and-forget on Save (the user doesn't wait), and awaited on the explicit Regenerate button (so the user sees the new summary on the next page render).

When `summary` is null (older rows or generation failed), both consumers fall back to their previous behavior. No data loss, no required migration.

## Generation prompt

```
You generate concise summaries for identity layers in a personal mirror system.

Rules:
- 2 to 3 sentences describing (1) the angle or domain in which it operates, (2) what it does and when it is activated, (3) what distinguishes it from other layers.
- Use neutral descriptive voice.
- Do not copy the prompt literally; synthesize.
- Match the language of the original layer content.
- Just the summary text, nothing else.
```

The user message body is the layer's content. Output is trimmed and capped at 500 chars before persisting.

## Validation

- 162 tests passing (11 new): 3 for `setIdentitySummary` covering write/overwrite/no-op-on-missing-layer, 7 for `extractPersonaDescriptor` covering summary preference, fallback, truncation, and the disambiguation case (the motivating bug between `tecnica` and `dba`), 1 confirming new layers start with `summary: null`.
- Manual validation deferred to the integration test pass with the user (alongside improvements #1 and #3).

## Design decisions

**Why fire-and-forget on Save (not awaited).** Same rationale as `generateSessionTitle`: the user shouldn't wait for a network round-trip to a model just to land on the Cognitive Map. If generation fails, the layer keeps its previous summary; consumers fall back to first line.

**Why awaited on Regenerate.** The Regenerate button is an explicit user action with the expectation that the new summary will be visible on the next page render. A short wait (~1-2s) is acceptable here; a stale summary after pressing Regenerate would be confusing.

**Why no migration script.** Older rows simply have `summary = NULL`. Consumers fall back gracefully. The user can populate any layer on demand via the Regenerate button. A bulk script could be added later if needed (e.g. as a one-time admin action).

**Why prefer `summary` over `content` first line.** The summary is by construction descriptive and disambiguating; the first line is structural (the prompt's opening sentence). For both card display and routing, descriptive beats structural.

**Why accept both `IdentityLayer` and `string` in `extractPersonaDescriptor`.** Backwards compatibility for any caller still passing raw content; callers in `reception.ts` and `adapters/web/index.tsx` are updated to pass the full layer.

## Risk

Low. Schema change is additive (nullable column). No data loss possible. Generation failures are silent and logged. Fallback to existing behavior is automatic when `summary` is null.

## Out of scope

- Bulk-regenerate-all-layers admin action. Mentioned as a possible future improvement; not needed now since the workshop has per-layer Regenerate.
- Showing the generated summary somewhere other than the workshop and the cards (e.g. in the Context Rail). The Rail uses `extractPersonaDescriptor` already, so it benefits transitively when the active persona has a summary.
