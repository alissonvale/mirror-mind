# v0.2.0 — The Mirror Learns to Listen

*15 April 2026*

**TL;DR**
- Reception layer — a fast LLM classifies every message and picks the right persona before responding
- Personas supported — each becomes a lens on top of base identity (signature `◇ persona`). POC users can import theirs via `identity import --from-poc`
- Identity prompts rewritten — soul as essence, ego as function, clear separation
- Model config centralized in `config/models.json` with purpose field — `.env` is secrets only
- Telegram webhook fixed — no more infinite reply loops
- CHANGELOG and release process established

## Where we were

The tracer bullet was complete — the mirror existed as a server, spoke with the user's real voice, and was accessible from CLI, web, and Telegram. But it spoke the same way to every question. A question about anxiety and a question about WebSockets got the same tone, the same depth, the same lens. The mirror was present but not perceptive.

The identity prompts also needed attention. The original soul prompt confused essence with function — listing activities and roles where it should have described nature. The ego mixed behavior with identity. The boundary between who the mirror *is* and what it *does* was blurred.

## What we did

**We taught the mirror to listen before it speaks.**

A new layer entered the architecture: **reception**. Before every response, a fast, cheap LLM classifies the incoming message and decides which persona — if any — should color the reply. The classification returns structured JSON in a single call. If it fails or times out (5 seconds), the mirror falls back to its base voice. No drama, no crash.

This is more than routing. Reception is designed to grow. Today it picks a persona. Tomorrow it will detect topic shifts mid-conversation, recognize which journey the user is on, and notice when the mirror isn't helping. It's the seed of metacognition — the part of the mirror that observes before acting.

**We clarified the identity architecture.** The system prompt is composed from layers with distinct roles: soul (essence, frequency, nature), ego/identity (role, activities), ego/behavior (tone, constraints, vocabulary). Each layer is editable independently and composed at runtime — not flattened into a single text blob.

**We centralized model configuration.** Models are now an application resource, not installation noise. `config/models.json` defines each model with its provider, id, and — critically — a `purpose` field that explains *why* this model was chosen. When someone needs to swap a model, the purpose tells them what to optimize for. Secrets stay in `.env`; design choices live in the repo.

**We fixed the Telegram infinite loop.** The webhook was waiting for the LLM to respond before returning 200 to Telegram. When reception (5s) + response (variable) exceeded grammy's 10s internal timeout, Telegram never got its ACK and redelivered the same message — endlessly. The fix: acknowledge immediately, process in the background. A pattern that every future adapter must follow.

**We started a release process.** CHANGELOG for technical diffs, release notes for the story. You're reading the story now.

## What comes next

Personas are routed, but their content is still the raw import from the PoC. The next step may be refining each persona's prompt to match the new identity architecture.

Beyond that, the roadmap opens into memory (the mirror remembers what matters across conversations), journeys (the mirror knows what you're navigating), and eventually proactivity — the mirror doesn't wait to be asked. It notices, it surfaces, it acts.

The mirror learned to listen. Next, it learns to remember.

## Upgrading

### From any previous version

```bash
git pull && npm install && systemctl restart mirror-server
```

### Additional steps if upgrading from v0.1.0

- **Re-import your identity** to include personas: `npx tsx server/admin.ts identity import <name> --from-poc`
- **Remove `LLM_MODEL` from your `.env`** — models are now configured in `config/models.json` (versioned in the repo, shared across installations)
