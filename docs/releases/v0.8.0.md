# v0.8.0 — The Mirror Knows Where I Am

*20 April 2026*

**TL;DR**
- New epic opens: **[CV1.E4 — Journey Map](../project/roadmap/cv1-depth/cv1-e4-journey-map/)**. The mirror gains situational awareness — **organizations** (broader context, a venture or community) and **journeys** (narrower context, a specific crossing) become first-class scopes with their own surfaces at `/organizations` and `/journeys`.
- Reception now returns **three independent signals** in one call: `{persona, organization, journey}`. The composed prompt becomes `soul → identity → persona → organization → journey → behavior → expression → adapter` — scopes cluster inside identity, broader before narrower, form cluster last.
- **Attachments are not owned by scopes — they will be linked polymorphically.** Decision registered ahead of CV1.E4.S2 so the same PDF can live under an organization and multiple journeys without duplication. Same principle will apply to extracted semantic memories when CV1.E3.S3 lands.
- **Four-surface model** formalized — **Cognitive Map** (structure), **Journey Map** (situational scopes), **Memory Map** (accumulated memory, planned CV1.E6), **Context Rail** (composition of this turn). Each answers a distinct question in a distinct temporal register.
- **Reception defaults to Claude Haiku 4.5.** Gemini Flash Lite couldn't hold 3-axis classification across ~17 candidates — sole-scope-in-domain cases leaked through even after three rounds of prompt engineering. Haiku is ~10× pricier per call but still trivial in absolute terms and restores routing quality.
- **Chat surfaces the routing signature** — each assistant message shows up to three badges (◇ persona, ◈ organization, ↝ journey), matching what composed the response. The rail already showed this for the current turn; badges anchor it to the message history.
- **Agentic turn deferred.** Task management was considered as the MVP for tool-use, but conversations with users showed unexplored value in the chat/prompt paradigm. CV1.E4 stays pure scope substrate; the agentic epic moves to the radar.

## Where we were

v0.7.0 tuned the mirror's voice — identity split into three ego keys, composition reordered, summaries flowing through cards and reception. But the voice is one axis. Anyone who uses the mirror across real life operates in multiple contexts at once: a venture, a personal crossing, a relationship with an organization. Before this release, the mirror knew *who* it was speaking as but not *where* the conversation was happening. Asking "quanto sobrou no caixa este mês?" and asking "quais prioridades da Software Zen?" pulled the same identity stack; the specific context was invisible to the model.

This release opens the second axis. The mirror now knows where you are.

## What we did

### Journey Map as a new conceptual frame

Before writing code, two concept docs landed — [`docs/product/journey-map.md`](../product/journey-map.md) and [`docs/product/memory-map.md`](../product/memory-map.md). They articulate the four-surface model and settle several foundational framings:

- **A scope is not an identity layer.** The original CV1.E5.S1 sketch placed organization as a new key in the `identity` table. That framing broke as soon as the scope needed to carry more than a briefing — situation that evolves, attached documents, filtered episodic memory. Scopes have structured metadata, lifecycle, and cross-cutting associations; they don't fit the prompt-as-text shape of identity. So they got their own tables and their own surfaces. CV1.E5.S1 is deleted as superseded.
- **An organization contains zero or more journeys.** Hierarchy is explicit (nullable FK on journeys). Personal journeys without an organization are a real and preserved case.
- **Attachments are first-class with polymorphic scope links.** When attachments land in CV1.E4.S2, they won't be owned by scopes; they'll associate with them through a polymorphic link table. The same PDF can live under `software-zen` and multiple journeys beneath it without re-upload.
- **Memory Map is the fourth peer surface.** Placeholder registered (CV1.E6) with draft stories. Naming the destination now means attachments and other memory mechanisms can be designed with a coherent surface in mind.

See the new decisions.md entries from 2026-04-20 for the reasoning around each.

### Scopes shipped across seven phases as one story

[CV1.E4.S1](../project/roadmap/cv1-depth/cv1-e4-journey-map/cv1-e4-s1-scopes-identity-routing/) delivered organizations and journeys end-to-end through the stack. Each phase left tests green; the story closed when manual validation against a real org + two journeys passed.

**Phase 1 — Schema + DB helpers.** Two symmetric tables: `organizations(briefing, situation, summary, status, ...)` and `journeys(briefing, situation, summary, status, organization_id nullable, ...)`. 27 unit tests cover CRUD, archive lifecycle, FK link/unlink, and `deleteUser` cascade through both. `deleteOrganization` unlinks journeys transactionally rather than cascading — the user's intent when removing an organization is "remove this organization", not "destroy its crossings", so journeys survive as personal.

**Phase 2 — Reception envelope expanded.** `receive()` now returns `{persona, organization, journey}` in one LLM call, each nullable, each validated independently. The prompt conditionally lists the three candidate sets when any exist; a capturing `completeFn` lets tests verify prompt structure without hitting a model. First draft of `evals/scope-routing.ts` written. Descriptor extraction for scopes lives in a shared `server/scopes.ts` (mirror of `server/personas.ts`).

**Phase 3 — Composition slots.** `composeSystemPrompt` accepts an optional `scopes: { organization?, journey? }`. Each scope renders as `briefing` followed by a delimited `Current situation:` block. Position in the prompt: `soul → identity → persona → organization → journey → behavior → expression → adapter` — the scope cluster sits inside the identity cluster, broader before narrower, with form rules preserving their recency weight at the end. Archived scopes never compose, even if their key is passed (a second layer of defense on top of reception's filtering).

**Phase 4 — `/organizations` surface + `/mirror/stream` wiring.** Create, edit, archive, unarchive, delete organizations. Summary pipeline extended with `generateScopeSummary` (branches for organization and journey). The chat stream starts threading reception's three signals into the composer, into the rail payload via `composedSnapshot`, and into assistant-entry meta (`_persona`, `_organization`, `_journey`). Other adapters (Telegram, CLI) were left at base composition on purpose — web is the reference implementation.

**Phase 5 — `/journeys` surface.** Symmetric with organizations, plus an organization selector and visual grouping by organization on the list page (personal journeys first, then one section per organization). Unknown `organization_id` on POST returns 400. Journey summary branch activates on save.

**Phase 6 — Rail scope lines + drawer scope dropdowns.** The rail's Composed block gains two new rows — organization and journey — each hidden by default, revealed when routing returns a key. The composed-prompt drawer gains two new dropdowns, and the shared drawer markup is extracted into `ComposedDrawer` (removes duplication between map.tsx and layer-workshop.tsx, which would otherwise have to change in lockstep for future axes). GET `/mirror` now derives all three composition axes from the last assistant entry's meta, so scope awareness survives page reloads.

**Phase 7 — Review pass + scope CSS + docs.** Story test guide (automated + manual happy path), refactoring notes separating applied from parked cleanups, worklog entry narrating the seven-phase arc, and scope-surface styles.

Coverage at close of the story: **237 passing**, zero regressions on top of the 162 from v0.7.0.

### Reception moves to Haiku — the prompt can only go so far

Manual validation caught a real miss: *"quanto sobrou no caixa este mês?"* activated the finance persona but not the `vida-economica` journey, despite the journey's descriptor covering exactly that domain. Three rounds of prompt engineering tried to recover:

1. **Persona and scope made explicitly complementary** — "a question about a domain activates both; persona gives voice, scope gives situational content; do not skip the scope because the persona already covers the topic".
2. **Replaced "don't activate on loose overlap" with positive criteria** — named, domain-matched, or sole-scope-in-domain.
3. **Made the sole-scope rule mandatory** — with a pre-return self-check: "If you find yourself about to return null for a scope, first check: is there exactly one scope whose descriptor covers the message's domain?".

All three landed on the reception prompt. Flash Lite still returned `journey: null`.

Swapping the reception model to `anthropic/claude-haiku-4.5` (same model as `main` since v0.7.0) fixed the routing immediately, on the unchanged prompt. The conclusion is architectural: scope routing across three independent axes and ~17 candidates exceeds what lite models can consistently handle. It's a capacity problem, not a prompt problem.

`config/models.json` updated accordingly. Cost rises from ~R$0.0002 to ~R$0.002 per reception call — ~10× more expensive but still trivial in absolute terms for single-user operation. Admins who need to cut cost can downgrade via `/admin/models` and accept missed scope routing; reception falls back silently on any failure, so a weaker model degrades gracefully rather than routing to the wrong scope.

Decision registered at [decisions.md — 2026-04-20 Reception defaults to Haiku 4.5](../project/decisions.md).

### Chat gets a routing signature per message

The rail already showed composition for the current turn. Badges add the same signal to each assistant message in the history — three compact pills above the bubble, one glyph per axis:

- **◇ persona** (warm beige) — the voice that responded.
- **◈ organization** (lavender) — the broader scope in play.
- **↝ journey** (sage) — the narrower scope in play.

Badges populate from the SSE `routing` event during streaming and from `_persona` / `_organization` / `_journey` meta for stored history. No axis shows when reception returned null — a meta-question response has no badges at all.

### Agentic turn deferred

Early in planning, task management was considered as the MVP for a shift into agentic behavior (the mirror does small things via tool use, not only responds). The conversation with users after v0.7.0 changed the direction: there is still substantial unexplored value in the chat/prompt paradigm itself. Forcing the agentic turn now would abandon a productive seam to chase a harder one. Task management stays on the radar as the first agentic MVP when the time is right; CV1.E4 stays pure scope substrate with no tool use.

See decisions.md 2026-04-20 *Agentic turn deferred* for the full reasoning.

### Development process — auto-commit per round

Small process update landed alongside the story. The implicit "commit in logical chunks" became explicit: every finished round of changes commits automatically, without waiting for instruction. A "round" = a finished phase, a coherent docs update, or a focused edit pass with tests passing. Push remains the user's explicit command — the gate is at remote visibility, where it matters, not at local history. Documented in [`docs/process/development-guide.md`](../process/development-guide.md#2-implement).

## What comes next

The epic continues. CV1.E4.S2 and S3 will populate the Journey Map surface with attachments and filtered memory:

- **CV1.E4.S2 — Documents attached to scope.** First use of the Attachments mechanism, chunked and embedded, polymorphic links to organizations or journeys. Briefing-from-attachments flow as a natural follow-up: upload material, the mirror drafts an initial briefing from its summaries, the user edits.
- **CV1.E4.S3 — Filter episodic and semantic memory by scope.** Sessions and extracted memories gain scope FKs so the journey detail page can show its own cut of memory; coordinates with [CV1.E3.S3 — long-term semantic memory](../project/roadmap/cv1-depth/cv1-e3-memory/).

After CV1.E4, the focus shifts to **[CV1.E3 — Memory](../project/roadmap/cv1-depth/cv1-e3-memory/)** — topic-shift detection (S1), conversation compaction (S2), and semantic memory extraction (S3). Then [CV1.E6 — Memory Map](../project/roadmap/cv1-depth/cv1-e6-memory-map/) as the surface that makes all of it legible.

The agentic turn remains on the radar. When it comes, the first MVP is task management bound to scopes — a bounded vocabulary of verbs (list, create, complete) with visible state changes as verification.

## Upgrading

### From v0.7.0

```bash
git pull && npm install && systemctl restart mirror-server
```

On first boot, new tables (`organizations`, `journeys`) are created via `CREATE TABLE IF NOT EXISTS`. No data migration, no schema alteration on existing tables, no user action required.

Reception model: existing installations keep whatever is currently configured in the DB (Flash Lite, unless changed). The seed change in `config/models.json` only affects new installs and the "revert to default" action at `/admin/models`. **Recommended for scope routing to work well:** change the `reception` role at `/admin/models` to `anthropic/claude-haiku-4.5`. See the release notes above for cost implications.

After upgrade, recommended (manual):

- Create organizations and journeys at `/organizations` and `/journeys` for the scopes you live in.
- Write briefing + situation for each; save; let the lite model generate summaries. The descriptors feed reception's classifier.
- Validate routing by sending a message that clearly belongs to one scope and watching the rail's Composed block + the message's badges.

### From older versions

Follow the v0.7.0 upgrade notes first (`ego/expression` layer, summaries, composed drawer, main → Claude Haiku 4.5 recommendation). This release stacks cleanly on top.
