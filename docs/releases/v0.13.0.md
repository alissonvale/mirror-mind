# v0.13.0 — The Family Moves In

*23 April 2026*

**TL;DR**

- **[Product Use Narrative](../product-use-narrative/)** — a new docs section under Product authoring four fictional users as a family of four. Each character is a complete, lived-in user: profile, identity (self/soul, three ego layers, five-to-six personas), two-or-three organizations, five-to-six journeys, and five sample conversations exercising different personas and scopes. **92 markdown files, ~3,600 lines of prose** in English, all written for both human reading (as docs pages) and machine loading (as fixture data).
- **`narrative load` admin command** — `npm run admin -- narrative load` provisions the four users into the database from the docs tree. Idempotent on every axis: users are kept across runs (same ID, same token), identity / orgs / journeys upsert in place, conversations dedupe by `(user, title)`. Real users (Alisson) are never touched. Every run writes a timestamped backup unless `--no-backup` is passed. Bearer tokens written once at first creation to a gitignored `.tokens.local` file. Companion `narrative tokens` prints the current map.
- **Sidebar gets a Docs link** above Logout (admin only). The route was always there; the entry point wasn't.
- **"Last conversation" cards become clickable.** The right-hand card on `/journeys` and `/organizations` rows was visible but not navigable. Now it's an anchor to `/conversation/<sessionId>` with a hover state matching the scope card on the left.
- **`/me` links its own forward-reference.** The "Export my data — coming with the Memory Map (CV1.E6.S6)" note now hyperlinks to the epic's docs page (admin only — docs are admin-restricted).
- **Dan Reilly is admin.** Coherent with the narrative — he's the family member who provisioned the server. The loader's `ADMIN_SLUGS` set is reconciled on every run.
- **513 tests passing**, unchanged from v0.12.0. This release is content-and-fixture-heavy; the new code paths inherit existing coverage.

## Where we were

v0.12.0 (*Taking Shape*) was driven by friction with one real user configuring one real production system. The work was responsive: a listing page that conflated browse with edit, a regenerate button with no visible feedback, sidebar ordering that didn't match daily focus. Each fix landed because someone using the system in earnest noticed a gap.

That responsiveness has limits. A single user, however serious, exercises a narrow path through the product. Multi-user features have stayed under-tested by definition. Demo material has been improvised: a screenshot here, a hand-typed example there, the assumption that the audience can fill in what's not shown.

This release is the deliberate move out of that posture.

## What we did

### A family of four

The Product Use Narrative is a new section of the docs under Product. It introduces a fictional family — Dan Reilly (48, IT infrastructure engineer in Lancaster PA, woodworker, homelabber), his wife Elena Marchetti (47, Associate Professor of English at Millersville, board member of the Lancaster Literacy Council), their son Eli (18, heading to Pitt for environmental science, organizer with a local climate coalition, birder), and their daughter Nora (16, sophomore, incoming editor of the school newspaper, runner with recurrent shin splints).

The framing is precise: **four parallel tenants on the same Mirror Mind server**. Narratively related but data-isolated — no shared organizations, no shared journeys, no cross-user sessions. The narrative reason they all run on the same host is Dan: he discovered Mirror Mind, provisioned a small VPS, and added the others as users.

Each character is fully authored:

- A **narrative profile** — bio, voice, what they carry, why they use the mirror.
- A **complete identity stack** — `self/soul`, `ego/{identity, behavior, expression}`, and five-to-six `persona/<key>` entries that fit the character (Dan: engineer, strategist, son, coach, chronicler, maker; Elena: scholar, essayist, teacher, advocate, mother, daughter; Eli: student, organizer, naturalist, writer, friend; Nora: journalist, editor, runner, reader, friend).
- **Organizations** — Dan's hospital, his homelab, his local meetup; Elena's university and the literacy council; Eli's incoming university and the climate coalition; Nora's high school and *The Trumpet*.
- **Journeys** — multi-month threads each character is actively living in: Dan's VMware-to-Proxmox migration, his mother's care, the college-money conversation he keeps deferring; Elena's second book, the literacy council's strategic plan, her father's aging, the empty nest incoming, the private essays she hasn't shown anyone; Eli's college transition, the breakup, finding his politics; Nora's first semester as editor, the running injury she's underreporting, the friendship with Amelia, her crush on Chase.
- **Five sample conversations** per character, each exercising different personas and scopes.

The whole thing reads as docs first — every file carries a breadcrumb back to the user hub, the user hub back to the family root, the family root back to the docs index. The content is fiction; the structure is the same the loader will eat.

### The loader

`npm run admin -- narrative load` walks the family tree and provisions everything. The contract is small enough to be obvious:

- Filename (without `.md`) is the **key** for identity rows, organizations, and journeys.
- Subpath under `identity/` is the **layer** (`self/`, `ego/`, `persona/`).
- Anything above the first `# ` heading in any file is **documentation chrome** — the loader strips it. Breadcrumb links live there. The same file works as a navigable docs page and a clean DB row, with no per-tool ceremony.
- Organization and journey files declare metadata in plain markdown — `**Status:** active`, `**Organization:** <key>` for journeys — followed by `## Briefing`, `## Situation`, optional `## Summary`.
- Conversation files use the existing canonical conversation format with frontmatter extended for `persona`, `organizations[]`, `journeys[]`.

Idempotency is the load-bearing feature. Every axis:

- **Users** are created on first run with generated bearer tokens. On re-runs they're kept — same ID, same token (stored hash unchanged). Roles are reconciled with the loader's `ADMIN_SLUGS` set (`dan-reilly` is admin, the rest are users), so a wrong-role user gets corrected on the next run.
- **Identity / organizations / journeys** upsert on `(user_id, layer, key)` or `(user_id, key)`. Re-running rewrites content in place — no duplicates.
- **Conversations** dedupe by `(user_id, title)`. To re-import (after editing the conversation files), pass `--reset-conversations`.
- **Real users are never touched.** The loader operates only on the slugs found under `docs/product-use-narrative/users/`. Alisson Vale's data is invisible to it.
- **Backup before write.** Every run (unless `--no-backup`) writes a timestamped copy of the SQLite file to `data/mirror.db.bak-narrative-<timestamp>`.

Tokens are persisted to `docs/product-use-narrative/.tokens.local` (gitignored). The DB stores only the SHA-256 hash. `npm run admin -- narrative tokens` prints the current map. If the file is lost, `--reset-tokens` regenerates them.

### Polish the populated system surfaced

The four users were loaded into the dev database in the same session this release was authored. Three small UI gaps became visible the moment more than one user existed in the system:

**The Docs surface had no entry point in the sidebar.** The `/docs` route is admin-only and was always there, but every visit required typing the URL. Sidebar footer now carries a Docs link directly above Logout, behind the same admin guard Admin Workspace uses.

**The "Last conversation" card on `/journeys` and `/organizations` was a `<div>`.** Visible. Informative. Not clickable. The shape of the page invited a click that did nothing. It now wraps in an `<a>` to `/conversation/<sessionId>` when a session exists, with a hover state matching the scope card on the left.

**The `/me` page's forward-reference to a not-yet-built feature was an inert label.** "Export my data — coming with the Memory Map (CV1.E6.S6)" is the kind of note the user reads, gets curious about, and has nowhere to follow. The CV1.E6.S6 label is now an anchor to the epic's docs page (admin only — docs are admin-restricted, so non-admins still see the plain text).

None of these were on the roadmap. Each was a real gap the populated system made impossible to ignore.

## Why this matters

The deliberate authoring of four characters does three things at once.

It produces **development data** that exercises features under realistic load. Multi-user scenarios, identity composition with real persona variety, organizations with narrative weight rather than placeholder names, journeys that span months, conversations with depth — all of this is now sitting in the dev database whenever you want it.

It produces **demo material** that can be walked through in presentations or onboarding without leaking real user data. Four characters with their own situations, voice, and use of the mirror. Anyone joining the project can read their way into the product through their stories.

It produces **design pressure**. Authoring a complete character forces the model to hold. The `maker` persona for Dan was added because the other personas didn't fit the woodworking shop — a gap the act of authoring exposed. Some conversations don't have a clean persona to tag with, which is itself information about where the routing layer's coverage stops. The `Memory Map S6` link was noticed because someone clicking through Eli's `/me` page realized the note was sterile. The narrative is a probe, and probes find things.

## What comes next

The populated system is the new baseline. Expected friction surfaces that were previously hypothetical:

- **Cross-user comparison surfaces.** With four users, `/users` (admin) becomes more useful — comparing how different users have configured their mirror is now a meaningful exercise.
- **Persona reception calibration at scale.** Reception's classifier was tuned against a single user's prompts. Four users with overlapping persona keys (each has a `friend`, two have a `writer`-shaped persona, etc.) will surface mis-routes the single-user setup couldn't.
- **Conversation surfaces under load.** With ~20 sample sessions across four users, `/conversations` and the per-scope listings get their first real volume of data to render. Performance, sort, filter ergonomics may shift.
- **[CV1.E4.S2 — Documents attached to scope](../project/roadmap/cv1-depth/cv1-e4-journey-map/)** — first use of the Attachments mechanism. Each character's organizations and journeys naturally invite documents (Dan's runbooks, Elena's draft chapters, Eli's lyrics, Nora's article drafts).
- **[CV1.E6.S6 — Export my data](../project/roadmap/cv1-depth/cv1-e6-memory-map/)** — now legibly linked from `/me`, more interesting to build now that the demo data has volume worth exporting.

## Upgrading

### From v0.12.0

```bash
git pull
npm install
systemctl restart mirror-server
```

**No schema migrations.** The narrative loader is data, not schema.

`package.json` bumps from `0.12.0` → `0.13.0`.

Recommended (manual, after upgrade):

1. **Provision the demo family** (dev only — do not run on prod with real users):
   ```bash
   npm run admin -- narrative load
   ```
   First run writes a backup, creates the four users, and prints the bearer tokens to `docs/product-use-narrative/.tokens.local`. Re-runs are idempotent — safe to run repeatedly while iterating on the narrative content.

2. **Browse the narrative** at `/docs/product-use-narrative/` to see what landed. Each user has a hub linking their files.

3. **Use one of the family members.** Pick a token from `npm run admin -- narrative tokens`, drop it in `~/.mirror/config.json`, and chat with the family's mirrors directly to feel the difference between user voices.

4. **Notice the small UI changes** while you browse:
   - Docs link in the sidebar footer (admin only).
   - "Last conversation" cards on `/journeys` and `/organizations` are now clickable.
   - `/me` links the CV1.E6.S6 reference (admin only).
