# v0.8.1 — Calibration and a path to zero cost

*21 April 2026*

**TL;DR**
- **Reception default swaps from Claude Haiku 4.5 to Gemini 2.5 Flash**, based on a three-model eval against the production DB. Flash matches Haiku's accuracy (9/11 on scope-routing) at ~3× lower per-token cost. `reasoning: "minimal"` is applied universally in reception — the option that closed the accuracy gap for Flash and is a pure win across all providers.
- **Reception gains latency logging and a thinking-block fallback** in the response parser. Ongoing model observability, and defensive resilience against providers whose JSON output ends up in reasoning blocks.
- **[Spike 2026-04-21 — Subscription-based LLM access via OAuth](../project/roadmap/spikes/spike-2026-04-21-subscription-oauth.md)** documents an investigation triggered by the question *"can we replace per-token billing with a subscription?"*. Consumer subscriptions across providers don't grant API access — but pi-ai supports OAuth against five subscription-backed provider paths out of the box.
- **[CV0.E3.S8 — OAuth credentials for subscription-backed providers](../project/roadmap/cv0-foundation/cv0-e3-admin-workspace/cv0-e3-s8-oauth-subscriptions/) is queued as the next priority story**, ahead of CV1.E4.S2 (attachments). Wires Google Cloud Code Assist free tier as the primary target — drops reception cost to zero at personal/family scale.

## Where we were

v0.8.0 shipped the Journey Map's first story — organizations and journeys as situational scopes, with reception routing across three axes. A post-release conversation opened two calibration questions:

1. Reception is the cognitive router. Today it picks persona + org + journey; tomorrow it may detect topic shifts, modulate response depth, or activate skills. What model should it actually be running? The v0.8.0 default (Haiku 4.5) was picked because Gemini Flash Lite couldn't hold 3-axis classification — but Haiku was the first upgrade that worked, not necessarily the best choice.
2. Per-token billing scales linearly with traffic. Are there subscription paths that would decouple cost from volume? Could the mirror piggyback on a Claude Pro/Max or equivalent?

Both questions were answered in the same session, with empirical evidence rather than speculation.

## What we did

### Three-model reception eval

`evals/scope-routing.ts` ran against the production DB (14 personas, 1 org, 2 journeys) three times, once per candidate model:

| Model | Score | Latency (min–max) | BRL/1M (in/out) |
|-------|-------|-------------------|------------------|
| Claude Haiku 4.5 | 9/11 (82%) ✅ | 1.3s – 2.6s | 5 / 25 |
| Gemini 2.5 Flash (w/ `reasoning: "minimal"`) | **9/11 (82%)** ✅ | 1.1s – 3.0s | 1.5 / 12.5 |
| Gemini 2.5 Pro | — blocked | — | 6.25 / 50 |

**The `reasoning: "minimal"` finding matters.** Without that option, Flash scored 8/11 — the model was over-activating scopes, inferring connections the prompt didn't ask for. With minimal reasoning, Flash becomes more decisive, more literal in following the sole-scope-in-domain rule. Lower cost, same accuracy, less variance. Applied universally in reception, including on Haiku (where it's a no-op; thinking is off by default) and on any future model plugged into the role.

**Gemini 2.5 Pro was blocked by a pi-ai parsing issue** specific to the OpenAI-compatible adapter reading OpenRouter's Gemini 2.5 Pro response shape. Direct `curl` against OpenRouter returned well-formed content; pi-ai returned `content: []` with `stopReason: "error"`. Parked until either pi-ai patches the path, or the OAuth `google-gemini-cli` provider (different path) unlocks Pro as a candidate.

### Subscription OAuth spike

A sibling investigation to the eval ran simultaneously: if per-token is expensive, is there a subscription path? The investigation traced the landscape and found:

- **Consumer subscriptions universally do not grant third-party API access.** Claude Pro/Max, ChatGPT Plus/Pro, Gemini Advanced, X Premium, Kiro, Cursor, Copilot — none of them. API remains per-token across all providers.
- **One exception path: OAuth against the provider's own tool subscription.** Claude Code does this against Claude Pro/Max. Codex CLI does it against ChatGPT Plus/Pro. GitHub Copilot extensions do it against the Copilot sub. Google Gemini CLI does it against Google Cloud Code Assist. The key observation: when a library wraps the same OAuth flow and exposes the authenticated token via a programmatic API, any app can ride the subscription.
- **pi-ai already does this.** Five OAuth providers are supported out-of-the-box: Anthropic, OpenAI Codex, GitHub Copilot, Google Cloud Code Assist, Antigravity. Each has a `loginProvider()` + `refreshToken()` pair plus a unified `getOAuthApiKey()` that the mirror would call at request time.
- **Google Cloud Code Assist for Individuals is the most attractive target.** Free tier for personal developer use covers the entire mirror's traffic at single-user/family scale. Quota (~60 RPM, ~1000 req/day on Gemini 2.5 Pro; higher on Flash) is more than enough. When exceeded, `GOOGLE_CLOUD_PROJECT` with billing transitions to paid Gemini rates seamlessly.

Operational pattern agreed: login on laptop (where the browser can handle OAuth normally), upload `auth.json` to the server via admin UI (or scp). Server refreshes tokens autonomously afterward. Dedicated Google account for the server, personal account stays on the laptop — clean isolation, independent free-tier quotas.

Full investigation at [spike 2026-04-21](../project/roadmap/spikes/spike-2026-04-21-subscription-oauth.md).

### CV0.E3.S8 queued as next priority

The spike produced a bounded implementation story: [CV0.E3.S8 — OAuth credentials for subscription-backed providers](../project/roadmap/cv0-foundation/cv0-e3-admin-workspace/cv0-e3-s8-oauth-subscriptions/). Scope:

- `oauth_credentials(provider PK, credentials JSON, updated_at)` table.
- `/admin/oauth` page with paste-from-laptop JSON upload per provider.
- `models.auth_type` column (`env` | `oauth`) added via additive migration.
- `resolveApiKey(db, role)` wrapper — routes through `getOAuthApiKey()` for OAuth providers, env-var for current behavior. Used by reception, summary, title, and the main response paths across the web and Telegram adapters.
- End-to-end validation via `scope-routing` eval pointed at `google-gemini-cli` / `gemini-2.5-flash`. Bonus: retry Gemini 2.5 Pro via the native provider path — may unblock the model that OpenRouter + pi-ai couldn't handle.

S8 moves ahead of CV1.E4.S2 (attachments) in priority. The cost-to-zero path and the model-eval context are both fresh; landing OAuth now preserves the option space for future reception capabilities.

## Upgrading

### From v0.8.0

```bash
git pull && npm install && systemctl restart mirror-server
```

No schema changes. No data migration.

Your existing install keeps whatever reception model is configured in the `models` table — the default change in `config/models.json` only affects new installs and the "Revert to default" action on `/admin/models`. To adopt the new default on an existing install, go to `/admin/models` and change the `reception` role's model to `google/gemini-2.5-flash` (provider `openrouter`, prices 1.5 / 12.5). Reception fires with the new model on the next request.

`reasoning: "minimal"` is applied universally — no action required. It's a no-op on Haiku; it closes an accuracy gap on Gemini 2.5 Flash; it protects future models from hiding JSON in reasoning blocks.

### From older versions

Follow v0.8.0 upgrade notes first (new tables for organizations/journeys, reception envelope changes, chat badges). This release stacks cleanly on top.
