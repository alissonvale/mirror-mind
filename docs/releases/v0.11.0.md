*22 April 2026*

**TL;DR**

Now `getOrCreateSession` resolves via `MAX(entry.timestamp)`, with `created_at` fallback for sessions that have no entries yet. Opening an old session to re-read no longer moves the user's anchor; sending a message in it does.
- **444 tests passing** (was 311 at v0.9.0). Three additive migrations across the release:`session_personas`, `session_organizations`, `session_journeys` (already present from v0.10.0 / CV1.E4.S4). No new tables this release — everything in v0.11.0 reuses the existing schema and meta-stamping.

## Where we were

v0.10.0 (*A Place to Land*) gave the mirror a home page and an editable scope-tag pool per session. The architecture was rich, but accumulated material wasn't surfaced anywhere — sessions piled up in the database with no way to browse them. Imported context from other AI tools couldn't even arrive: years of conversation in Gemini stayed there because there was no path in.

This release is about **legibility of accumulated material**. What the mirror carries — past conversations, by scope, across scopes, from prior tools — becomes visible and navigable.

## What we did

### CV0.E3.S9 — Import conversation history from markdown

Admin CLI command:

```bash
npx tsx server/admin.ts conversation import <user> \
  --dir <path> \
  --persona <key> \
  [--organization <key>] [--journey <key>] \
  [--dry-run]
```

Each markdown file becomes one session; alternating `**User:**` / `**Assistant:**` blocks become entries in order. Frontmatter `title` becomes session title (filename as fallback). The canonical format is documented at [`docs/product/conversation-markdown-format.md`](../product/conversation-markdown-format.md) as a public artifact — anyone writing a converter for ChatGPT, Claude, custom logs only needs to produce files matching the contract.

`--dry-run` parses, validates persona/org/journey resolve, lists what would be created, and exits without writing. Per-file failures (alternation violations, empty bodies) record into the report and don't abort the run; one bad file in a batch doesn't block the rest.

Imported assistant messages are stamped with `_persona` / `_organization` / `_journey` meta so the existing aggregations treat imported sessions identically to organic ones.

The first real use migrated 27 conversations from `~/Code/szen_mind/conversas/zenith-estrategia-gpt-4o/` into the production mirror as sessions tagged `estrategista` + `software-zen`. The personal runbook for the migration lives in the user's local `mirror-poc/runbooks/` (out of scope here — operational, not project doc).

### CV1.E4.S5 — Scope page becomes an ateliê

`/organizations/<X>` and `/journeys/<X>` previously rendered briefing + situation + a single "Last conversation" card. Now they show a Conversations section with the 5 most recent sessions tagged to that scope (title, persona badge, relative time, 2-line preview). When `total > 5`, a quiet "View all N conversations →" link points to `/conversations?organization=<key>` (or `?journey=<key>`).

Click on a row → `/conversation/<sessionId>` (S5's route) → loads the named session for continuation.

The trim to 5 was a follow-up — initially the page rendered all sessions, which overwhelmed the workshop once a scope accumulated more than a handful (S9 made this acute the same day with 27 imports). The teaser-plus-link pattern keeps the workshop scannable while preserving full discoverability.

`getOrganizationSessions` and `getJourneySessions` accept an optional `limit` and return `{ rows, total }`. The route handlers pass `limit=5`; other callers can omit for everything.

### CV1.E6 activated — S1 Conversations browse

Memory Map epic activates with its first concrete piece. New top-level surface at `/conversations`:

- Filter dropdowns: persona, organization, journey (single value each, "All" default). Selected values survive bookmarking via query string.
- Sort: recency (last activity, same definition as the home Continue band).
- Pagination: 50/page, "Show N more" anchor preserves filter params.
- Each row: title link, persona / org / journey tag pills (the cross-scope view needs scope visible inline), relative time, 2-line preview.
- Active session: marked with a "current" badge so the user can locate their thread at a glance.
- Empty states with voice: "No conversations yet" (no data) vs "No conversations match these filters" + Clear filters affordance (filtered narrowly).

URL is `/conversations` top-level, not under `/memory` — until a Memory Map landing exists (future S7), the standalone surface reads cleaner. The eventual `/memory` becomes the meta-surface that ties this to other sections (attachments, insights).

The original CV1.E6 plan had S1 as "landing + section cards." Reordered: S1 became Conversations browse (was draft S3 "Episodic browse"), and the landing demoted to S7 "future" — driven by S5's revealed need for a destination for "View all (filtered)" rather than a meta-surface that has nothing yet to point to.

### Sidebar restructure (CV0.E4.S9)

The Conversation entry got promoted from a single link to a section, peer of the other sections in the sidebar:

```
Conversation        (section)
  Current           → /conversation
  See All           → /conversations

What I'm Doing      (section)
  Journeys          → /journeys
  · O Espelho       → /journeys/o-espelho       (third level)
  · Vida Econômica  → /journeys/vida-economica
  ...

Where I Work        (section)
  Organizations     → /organizations
  · Software Zen    → /organizations/software-zen
  ...

Who Am I            (section)
  Psyche Map        → /map
```

The third level (the actual journey/organization names) lists active scopes only; archived scopes don't crowd navigation. Visually differentiated: smaller font (0.78rem), dimmer color (#888), `·` marker on the left, slightly tighter padding — reads as "children of" without needing a label.

The sidebar nav now scrolls (overflow-y: auto, min-height: 0) so the footer (Admin Workspace + Logout) stays pinned to the bottom even when nested entries fill the visible area. Subtle dark scrollbar matches the sidebar's tone.

### Correction — current session = last activity

Phase 2 of CV1.E4.S5 introduced `markSessionActive` that bumped `sessions.created_at` when a session was opened via `/conversation/<sessionId>`. The intent: make the named session "active" for the next default `/conversation` resolution.

Real use revealed the conflation. The user opens an old session in `/conversations` to re-read context, then clicks back to `/conversation` planning to continue their actual work — and lands on the re-read session instead. *Reading is not continuing.*

Fix: `getOrCreateSession` resolves via `MAX(entry.timestamp)` with `created_at` fallback. `markSessionActive` deleted. Opening doesn't change current; sending a message in the opened session naturally updates activity, and that session becomes current.

## Upgrading

### From v0.10.0

```bash
git pull
npm install
systemctl restart mirror-server
```

No schema migrations. v0.11.0 reuses the tables CV1.E4.S4 introduced (`session_personas`, `session_organizations`, `session_journeys`). Everything else is meta-stamped on assistant messages or derived at query time.

`package.json` bumps from `0.9.0` → `0.11.0` — v0.10.0's bump was deferred (the tag was retroactive), so this release catches both.

Recommended (manual, after upgrade):

1. **Visit `/conversations`** to see the new browse surface. Filter by org/persona/journey to narrow.
2. **Visit `/organizations/<your-org>`** to see the trimmed Conversations section + "View all" link.
3. **Look at the sidebar** — *Conversation* now has *Current* and *See All* under it; active journeys and organizations appear as nested entries.
4. **Try the conversation import** if you have prior history elsewhere. Format spec at `docs/product/conversation-markdown-format.md`.
