# v0.25.0 — Choosing how the mirror speaks back

*5 May 2026*

**TL;DR**

- **CV1.E15 epic — Model configuration per scope** (7 stories shipped). The global `main` model becomes a per-scope choice. Resolver chain `turn → session → scene → global`; every assistant turn stamps the resolved model.

The picker's option value is the bare `model_id` (no provider prefix); the label exposes provider for clarity when curated entries duplicate an OpenRouter row.

Pouch summary appends `· <model_short>` when an override is set.

Every assistant entry now stamps `_model_provider` + `_model_id` in meta — `/admin/llm-logs` shows the actual model that ran each turn (logUsage + logLlmCall both use the resolved values, not the global default).
  - **S5 — Per-turn kebab menu (admin).** `×` becomes `⋯` for admins; the panel lists "Re-executar com modelo…" (assistant turns only) + "Excluir" (preserves the existing forget-turn POST). Non-admins keep the bare `×`. Single rerun popover rendered once per page (admin only); opened by `[data-rerun-trigger]` clicks across any bubble.
  - **S6 — Destructive rerun endpoint.** `POST /conversation/turn/rerun` (admin-only) replays the user message that produced an assistant entry through a chosen model and **mutates the assistant entry in place** — same id, new content, new `_model_provider/_model_id`, plus `_rerun_at` timestamp. Inherits persona/scope/mode/length/Alma flag from the parent meta — only the model changes. Reception is NOT re-run. Pi-ai shape fields (`provider`/`model`/`usage`) replaced too so the entry tells the truth about its current text. `logLlmCall` writes against the same `entry_id` for trace.
  - **S7 — Per-session toggle for model badges.** Shipped first as "auto-show on divergence", rewritten same-day after manual testing surfaced that the rule was hard to predict and the rail's "model" line still lied when the session mixed. The `done` SSE event carries `assistantModel: { provider, id }` so streamed bubbles paint the badge live without reload.
  - **Per-model rail breakdown.** `SessionStats.model: string` (always the global default, lying when the session mixed) retires; `SessionStats.models: Array<{ model_id, count }>` reads from the stamped `_model_id` meta. ## What's parked for next session

**Cross-provider auth resolution.** `resolveApiKey(db, "main")` still resolves through the global main role's `auth_type`. When the admin pins a different OpenRouter model as session/scene/turn override, env-based auth keeps working — the provider doesn't change. But a cross-provider override (e.g. global=openrouter, session=anthropic-direct OAuth) would need a richer auth resolver that picks `auth_type` per-provider. Documented in the S4 doc; not blocking for personal/Veronica use.

**SSE streaming for rerun.** The destructive rerun endpoint is synchronous JSON + page reload today. The popover stays open during the call and shows `"Composing…"` (same hardcoded microcopy the main chat uses in `chat.js:1267-1268` — itself an i18n debt). To eliminate that duplication, the rerun would emit `routing/status/delta/done` events the way `/conversation/stream` does, the popover would close on submit, and the targeted bubble would repaint in place reusing the chat's own render pipeline. ~2-3h refactor; benefit marginal until the rerun surface is used heavily.

**`session-stats` cost using the resolver.** Cost estimate in the rail still uses the global model's per-1M prices — minor inaccuracy when the session mixes models with different prices. Doesn't affect generation. Deferred — the per-model breakdown row tells the truth visually; the cost line is a rough estimate either way (chars/4 token approximation).

**Live toggle on `show_model_badges` without reload.** Saving the toggle currently triggers a native form submit + reload. A `refreshConversationHeader`-style async + DOM walk would work but isn't worth the code on a low-frequency toggle.

**i18n the chat status microcopy.** `chat.js:1267-1268` ("Composing", "Finding the voice") and the rerun popover's `"Composing…"` should live behind a single i18n key surface. Trivial once the rerun follow-up above forces both sites through one render path.

**Lazy materialization of sessions.** The orphan cleanup is opportunistic (1-min window). The cleaner design — `createFreshSession` returns a UUID without persisting; `appendEntry` materializes on first append — was deferred because it requires touching `appendEntry`'s signature across ~15 call sites. Revisit if the 1-min window ever causes confusion in the wild.

## Where we were

v0.24.0 (*The mirror reads itself*) flipped every entity surface from edit-by-default to read-by-default. The user could now navigate the territory of personas / journeys / orgs / scenes as a memoir and edit only by intent. The metabolism of *reading* was solved.

What carried over was **operational opacity**: the mirror had become a single-model creature again. Every turn went through the same global `main` model regardless of context — a journaling cena and a strategic deliberation cena ran on the same model whether the budget allowed it or not. The admin could change the global, but it leaked into every other session. There was no per-conversation lever, and no way to A/B a turn against a different model without starting a new session.

A second pressure: the rail's "model" line claimed a single model regardless of what actually ran. After CV1.E13's "what's actually here" treatment of the territory, having the inspect surface lie about the runtime model felt like a leftover from before the read-by-default era.

CV1.E15 set out to give the admin the lever and the visibility, in that order.

## What we did

### CV1.E15 — Model configuration per scope

The story sequence (S1 → S7) was deliberately designed so each story was independently shippable (or at least independently testable) and so the *user-visible behavior* depended on S4 — the resolver — landing. S1-S3 wrote columns and surfaced UI but didn't change generation; S4 wired the resolver chain into every main path; S5-S7 added affordances on top. The reverse order (S5 first) would have shipped a UI that did nothing visible.

**S1 — Catalog + picker** is the substrate every other story consumes. The decision to use a native HTML `<datalist>` combobox (over a JS-driven autocomplete library) was deliberate: zero new dependencies, both keyboard and click work for free, fallback to bare text input when the catalog is empty. The OpenRouter catalog is fetched on-demand and cached for an hour; when fetch fails (no key, network down), the curated extension carries on alone with no error surface. Pricing converts USD → BRL using the existing `usd_to_brl_rate` setting so the picker labels read in the local currency. The first manual test surfaced two bugs: the option value carried `provider/model_id` doubled (because OpenRouter's `model_id` already includes the upstream vendor prefix) and `/admin/models` rendered flush-left full-width while every other admin surface centered to 1080px. Both fixed in the same commit, with a generic `.admin-page` class so the next admin surface that needs centering can adopt it.

**S2 — Per-scene model override** added two nullable columns to `scenes` and the picker inside `<details class="cena-advanced">`. Decision: the field renders ONLY when `user.role === "admin"` AND the route handler ignores model body params for non-admin requests. Two layers of gating because a single one is brittle — the form could be hidden but the route still vulnerable, or vice versa.

**S3 — Per-session model override** mirrored S2 at the conversation surface. Header pouch picks up an admin-only "Modelo" row with the same picker; pouch summary appends `· <model_short>` when set, so the admin sees the override status without expanding the pouch. POST `/conversation/model` returns 403 for non-admin so any UI surface that posts there is forced to gate by role at render time.

**S4 — Resolver + per-turn stamping** is the keystone. `resolveMainModel(db, sessionId, userId)` returns `{ provider, model, source, globalConfig }`. The `source` field (`"session"` | `"scene"` | `"global"`) feeds S7's later badge logic and the `/admin/llm-logs` row. Each tier requires BOTH provider and id; partial values fall through (an admin who fills one column without the other doesn't accidentally orphan the model). Plumbed through `server/index.tsx` (API), `adapters/web/index.tsx` (web SSE main streaming + divergent runs), `adapters/telegram/index.ts`. Every assistant entry now stamps `_model_provider` + `_model_id`; `logLlmCall` and `logUsage` use the resolved values so the audit trail tells the runtime truth.

The auth resolution caveat is documented but not fixed: `resolveApiKey(db, "main")` resolves through the global main role's `auth_type`. Same-provider overrides (the common case — admin pinning a different OpenRouter model) work unchanged because env auth doesn't depend on the model id. Cross-provider overrides would need a richer auth resolver. Recorded as parked.

**S5 + S6 + S7** shipped together because they're mutually dependent on the surface level. The kebab menu (S5) opens a popover that calls the rerun endpoint (S6); the badge (S7) reads the model the rerun stamped. The single popover instance rendered once per page (admin only; non-admin renders skip both the triggers and the popover) opens on `[data-rerun-trigger]` clicks via event delegation, so it survives bubble re-renders without re-attaching listeners.

**S6's destructive shape** (vs the originally-drafted side-by-side variant in CV1.E8.S2, which this epic supersedes) was the user's explicit ask. When a turn lands wrong, the obvious move is to replace it. Side-by-side `divergent_runs` plumbing serves persona/scope branching well (CV1.E7.S8) but doesn't fit "same prompt, different model" — there's no comparison value once the user has chosen the rerun. The rerun mutates the assistant entry in place: same id, new content, new `_model_provider/_model_id`, plus `_rerun_at` timestamp for trace. Reception is NOT re-run; persona/scope/mode/length/Alma flag inherit from the parent meta. The pi-ai shape fields (`provider`/`model`/`usage`) get replaced too so the entry's top-level `provider`/`model` field tells the truth about its current text rather than the original.

**S7 was rewritten same-day** after manual testing. First take: render `⊕ <model>` on bubbles whose stamped `_model_id` differed from the session's currently-resolved model. The rule was hard to predict in practice (badge appeared and disappeared as the resolver chain shifted under the admin's feet — switch the session override mid-thread and every prior turn lit up; switch back and they all went quiet). The user named it: *"está meio confuso essa coisa de quando exibe ou nao o modelo do turno."* And the Look-inside rail's "model" line still claimed a single model. Wrong primitive twice over.

The rewrite: a binary toggle inside the Advanced pouch (`ocultar / mostrar`). Persisted per-session, default off. The admin enables it in the conversation where they're actively comparing, and it stays on across visits to that session without leaking into other sessions where the comparison isn't relevant. In parallel, the rail's "model" line stops claiming a single model and reads as a per-model breakdown.

The folder slug `cv1-e15-s7-divergence-badge/` stays put for git history continuity even though the story is no longer about divergence.

### Alma's expression bypass becomes conditional

Reported during testing: a session inherited `conversational/standard` from a cena, the admin switched to `conversational/brief` in the header, but responses kept landing long.

SQL forensics:

```sql
SELECT id, response_mode, response_length FROM sessions ...
→ conversational | brief

SELECT json_extract(data, '$._mode'), json_extract(data, '$._length'), ...
→ conversational | brief | (assistant text...)

SELECT role, length(response), error FROM llm_calls WHERE entry_id = (...)
→ main | 1393 | (no expression row)
```

Plumbing was perfect: the toggle wrote to `sessions`, the entry stamped `_length=brief`. But only `main` ran; no `expression` row in `llm_calls`. Root cause: the cena had `voice='alma'` and the session inherited it. The streaming pipeline's `if (isAlma) reply = draft` bypass — designed to preserve the Alma's "acolher → iluminar → revelar" prose verbatim — was unconditional, so the explicit header overrides were silently dropped.

Fix: bypass only when there's NO explicit override.

```ts
const skipExpression = isAlma && modeOverride === null && resolvedLength === null;
if (skipExpression) {
  reply = draft;
} else {
  // run expression with the chosen mode/length
}
```

The original Alma behavior (preserve prosa verbatim) holds when the admin doesn't configure overrides — the bypass yields only when the admin is explicitly taking control. Same logic applied symmetrically in the rerun endpoint, reading the parent's `_mode_source` (`"session"` means override) and `_length` (non-null means override).

### Empty-session orphan cleanup

The user observed: "estamos com muitos órfãos no banco de dados." Diagnosis: `createFreshSession` (begin-again, save-and-start, `/inicio`) and `getOrCreateSession` (first `/conversation` visit) both eagerly insert the session row, and `forgetTurn` deletes entries but not the row when it empties. The list views (`listRecentSessionsForUser`, `getConversationsList`) papered over with `INNER JOIN entries`, but the DB kept growing.

Three layers of defense:

1. **`forgetTurn`** drops the session row in the same transaction when the last entry leaves (with junction cascade — `session_personas`, `session_organizations`, `session_journeys`).

2. **New `pruneEmptySessionsForUser(db, userId, opts)`** sweeps that user's zero-entry sessions older than `minAgeMs` (default 1 min — race protection for "user just opened the page and is typing"). Cascade junctions; idempotent.

3. **`createFreshSession`** and **`getOrCreateSession`** invoke the prune before they insert/read. Each begin-again or save-and-start cleans up the previous abandoned draft; each `/conversation` visit cleans up drafts that aged out.

Plus a one-shot migration sweep at `openDb` time:

```sql
DELETE FROM sessions WHERE id NOT IN (SELECT DISTINCT session_id FROM entries);
```

(with junction cascade). Idempotent — boots after the new code paths shipped find no rows to delete.

Trade-off vs full lazy materialization (where `createFreshSession` returns a UUID without persisting and `appendEntry` materializes on first append): not real-time. Up to 1 min window between abandonment and cleanup. Acceptable — eliminates accumulation continuously without touching `appendEntry`'s signature across ~15 call sites.

### Polish + UX cleanup

After CV1.E15 closed, manual smoke surfaced six visual / UX issues, each fixed in its own small commit:

1. **Chat shell padding.** `.chat-shell` sat directly inside `.topbar-main-wide` (which has `padding: 0`), so the header, messages, and input row all touched the viewport edges. Added padding (1rem top, 1.5rem sides, 1.5rem bottom on desktop; 0.75rem mobile) and adjusted the height calculation accordingly.

2. **Opaque kebab menu.** The `⋯` menu rendered translucent when open. Root cause: `opacity: 0.7` on `.msg-actions` (the `<details>` parent) propagated to the panel — CSS opacity always propagates to descendants, regardless of background colors. Moved opacity to the trigger only; panel renders fully opaque whenever open.

3. **Advanced zone label.** When a cena pre-set mode + length, the Advanced pill read e.g. `ensaísta/padrão ▾` with no surrounding context, while every other header zone (ELENCO, CENA, CONTEXTO) showed its label first. The user reported "the Advanced doesn't appear when there are no messages" — actually it was there, just unrecognizable without the label. Added `<span class="header-zone-label">AVANÇADO</span>` before the pouch; pill summary changes from "Avançado" → "auto" when both axes are auto (the zone label already says it).

4. **Advanced panel anchors left.** With the new label pushing the pouch's typical position rightward, the panel (anchored `right: 0`) escaped past the viewport's left edge when the pouch wrapped to the start of a new header row. Switched to `left: 0; right: auto` + `max-width: calc(100vw - 1rem)` as a safety net.

5. **Streamed bubbles get the admin kebab live.** When the user sent a message, `chat.js` streamed the new bubble and called `attachDeleteForm` to wire its per-turn affordance. The function unconditionally rendered the simple `×` — admin saw `×` on the bubble that just streamed and `⋯` on every prior bubble (SSR), with the kebab "appearing" only after a manual reload. Branched on `isAdminViewer()` (presence of `#rerun-popover` in the DOM as the cheapest reliable signal client-side). Localized microcopy exposed via `data-i18n-*` on `#messages` so the streamed markup matches the SSR-localized strings.

6. **Rerun popover locks during pending.** During the in-flight POST to `/conversation/turn/rerun`, the user could mash Submit twice (firing a second rerun mid-mutation) or click Cancel / outside-click and dismiss the popover while the server was still working — the reload then landed on inconsistent state. Added a `pending` flag + `setPending()` helper that disables both buttons + both inputs and blocks `closePopover` until the call completes (or errors out). Status text borrowed from chat.js' own pipeline microcopy ("Composing…") so the rerun reads as a continuation of the same generation flow.

A separate small fix earlier in the cycle (`f979565`) made the rail's live-update path read localized strings via `data-i18n-*-line` attributes — `chat.js` was overwriting SSR-rendered Portuguese ("cena: foo") with hardcoded English ("scene: foo") after every turn.
