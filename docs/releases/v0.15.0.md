# v0.15.0 — A voice, a cast

*24 April 2026*

**TL;DR**

- **[CV1.E7.S5 — Multi-persona per turn (integrated voicing)](../project/roadmap/cv1-depth/cv1-e7-response-intelligence/cv1-e7-s5-multi-persona/)** activates. `ReceptionResult.persona: string | null` migrates to `.personas: string[]`. Zero, one, or multiple lenses per turn.
- **Composer renders multiple persona blocks simultaneously**, preceded by the shared instruction *"Multiple persona lenses are active simultaneously. Speak with one coherent voice that integrates all of them; do not label segments."* Single-persona behavior is identical to v0.14.0.
- **Expression pass receives the list** and reminds the model: one unified voice, preserve each lens's distinctive contribution to the draft, no segment markers inside the text.
- **Set-based bubble signature.** `computeBubbleSignatures` now tracks the previous assistant's persona *set*. Each turn's rendered badges are the personas new relative to the previous turn. `[A, B]` followed by `[B, A]` produces no fresh badges — the comparison ignores order. Color bar uses the primary persona (first in the list).
- **Streaming routing event emits both shapes** — `personas: string[]` + `personaColors: Record<key, color>` as canonical, `persona` + `personaColor` as backward-compat scalars so any out-of-tree consumer keeps working.
- **Meta persistence dual-shapes.** Assistant entries stamp both `_personas` (array, canonical) and `_persona` (first element, backward-compat) — every existing consumer of the singular keeps reading it; new consumers can migrate to the array at their own pace.
- **All three adapters migrated**: web (SSE routing event + meta), Telegram (composer + expression + reply signature listing all personas on one row), API (`POST /message` response body now carries `personas`).
- **640 tests passing**, up from 627 at v0.14.0. +13 new across reception (4 — plural shape, legacy wrap, dedupe, drop unknowns), composer (4 — empty/single/multi/unknown filtering), expression pass (1 — multi-persona block), and UI multi-persona bubble signature (5 — first-turn badges, same-set reorder, same-set repeat, add-one delta, primary color bar).

## Where we were

v0.14.0 (*Finding the voice*) installed two things: the pipeline-over-prompt pattern (S1 — expression as a post-generation pass) and the UI scaffolding for a cast that would form across turns (S2 — conversation header + slim rail). The cast rendered as avatars in the header; the bubble signature (color bar + transition badge) was already set-aware in its data shape. The forward-compat was deliberate.

What was missing: reception still returned `persona: string | null`. No matter how rich the UI was prepared to become, each turn only activated one lens. When the user asked a question that genuinely spanned two domains — *"como pensar a estratégia de divulgação?"* deserves both a strategist and a communicator — the mirror could only bring one.

This release fills that gap for the integrated voicing case.

## What we did

### One contract, array-shaped

`ReceptionResult.persona: string | null` → `.personas: string[]`. Zero, one, or more keys per turn. Reception's prompt was rewritten to ask for an array with explicit minimum-sufficient-set guidance:

> Prefer the minimum number of personas. Return more than one only when a single persona cannot carry the answer's full substance without the model acting outside its descriptor.

The leading lens goes first — it's the one whose color drives the bubble's color bar, and whose voice frames the opening of the reply. Order is preserved from reception through the composer, the expression pass, and the meta stamp on the assistant entry.

### Composer speaks as one voice

When the array has one element, the composer renders identically to v0.14.0's singular path — no prefix, one block. When it has two or more, the composer prepends:

> Multiple persona lenses are active simultaneously for this turn. Speak with one coherent voice that integrates all of them — each lens contributes its depth to the reply, but the voice is unified. Do not label segments or mark transitions between lenses inside the text; weave them into a single answer.

The user sees one reply whose depth comes from multiple lenses cooperating, not two stitched-together sections. *"qual seria a estratégia de divulgação do espelho para o público da Software Zen?"* produces a single strategic-and-executional answer that reads as one voice integrating estrategista's positioning with divulgadora's channel + copy thinking. Segmented voicing (explicit `◇ X ... ◇ Y ...` inside the text) is **not** this release — parked for the follow-up story S5b.

### Expression pass keeps the voice unified

The expression pass (the second LLM call that shapes the draft per response mode) now gets `personaKeys: string[]` and adds a reminder to its system prompt when the list has 2+:

> Multiple persona lenses produced this turn together: "estrategista", "divulgadora". The draft integrates all of their contributions into one unified voice. Preserve each lens's distinctive contribution to the draft; do not label segments or mark transitions between lenses inside the text. Your job is form only — the integration is already done.

This locks the no-segmentation contract into both composer-stage and expression-stage prompts.

### Bubble signature goes set-based

`computeBubbleSignatures` (the server-side walk that decides what to render on each assistant bubble) now tracks the previous assistant's persona *set*, not the singular. Per-turn output changes from `{ showBadge: boolean, persona: string | null }` to `{ newPersonasThisTurn: string[], primaryPersona, personas }`. One `◇ key` text badge renders per entry in `newPersonasThisTurn`. Consequences:

- Turn 1 with `[estrategista, divulgadora]` → two badges.
- Turn 2 with same set → zero new badges; color bar continues.
- Turn 2 with `[divulgadora, estrategista]` (reordered) → still zero new badges (set comparison, not order).
- Turn 2 adds `mentora` → one new badge.
- Persona-less turn between two persona'd turns resets the tracker; the next persona'd turn's entire set counts as "new" again.

The color bar uses the primary persona's color (first in the list) — a gradient or dual-tone bar for multi-persona bubbles is parked for S5b.

### Streaming event carries the plural

The SSE `routing` event in the web adapter emits:

```json
{
  "type": "routing",
  "personas": ["estrategista", "divulgadora"],
  "personaColors": {
    "estrategista": "#b88a6b",
    "divulgadora": "#8aa08b"
  },
  "persona": "estrategista",
  "personaColor": "#b88a6b",
  ...
}
```

The plural is canonical; the singular scalars stay for backward-compat with any client that reads them. The client-side helpers — `attachPersonaSignature`, `ensureCastAvatar` — accept both shapes and normalize internally.

### Meta persistence dual-shapes

Assistant entries persisted during a turn now carry both:

- `_personas: string[]` — the new canonical field.
- `_persona: string | null` — first element of `_personas`, or null when empty.

This lets every current consumer of `_persona` keep working without changes (`/conversations` filter, me-stats "most active persona," scope-sessions "last persona," `getPersonaTurnCountsInSession`). Downstream migrations to the plural can happen one consumer at a time, or ride on S5b when richer data requires it. Historical sessions whose meta only carries the singular still render correctly — every reader path checks the array first and falls back.

### Adapters migrated in lockstep

- **Web** — `/conversation/stream` handler loops over `reception.personas` for cast auto-seed, passes the array to composer and expression, emits the plural routing event, and stamps dual-shape meta on the assistant entry.
- **Telegram** — same pipeline flow; reply signature line renders every persona on one row: `◇ estrategista ◇ divulgadora\n\n<reply>` (single persona behaves identically to before).
- **API (`POST /message`)** — response body gains `personas: string[]` alongside the compat `persona: string | null`.

## Why this matters

v0.14.0 said *form finds its voice*. v0.15.0 says *the cast that produced the form is itself plural*. One persona was already a stretch from "the mirror" singular — two personas is the honest admission that a person's answer to a real question often needs more than one lens to be correct.

The integrated voicing default is the quiet version of this. A user asking *"como pensar isto?"* doesn't want the mirror to parade perspectives; they want one coherent answer that carries depth. But the code path opened in v0.15.0 is also what makes the segmented case (S5b) possible without another big migration — reception is already returning arrays, composer and expression already think in plurals. The next story is a prompt-engineering + UI-render story, not a contract-reshape story.

### What does NOT change for users upgrading

- Single-persona behavior is bit-for-bit identical. If reception would have returned `"mentora"` in v0.14.0, it now returns `["mentora"]` — the composer renders the same block, the expression pass produces the same draft, the UI renders the same single badge.
- Sessions created before this release still render correctly via the `_persona` fallback path.
- Reception's classifier still uses the same model and timeout.
- Rail (admin-only, Look inside) still shows the primary persona in its Composed snapshot — multi-persona surfacing in that section is future work.

## What comes next

- **S5b — Segmented voicing (proposed).** The opt-in complement to S5's integrated default: when the user asks for distinct perspectives, or when two lenses disagree and the tension is the answer, reception classifies `voicing: "segmented"` and the expression pass emits explicit `◇ X ... ◇ Y ...` blocks inside the reply. Each segment renders with its own color bar (or stripe). Small scope.
- **[S3 — Conditional scope activation](../project/roadmap/cv1-depth/cv1-e7-response-intelligence/).** Orgs and journeys stop composing the prompt "always if tagged" — only when the turn genuinely touches the scope's territory. Prompt enxuga in fundo-not-foreground moments.
- **[S4 — Conditional identity layers](../project/roadmap/cv1-depth/cv1-e7-response-intelligence/).** `self/soul` and `ego/identity` enter the prompt only on turns that invite existential depth. Small talk loses the existential overhead.
- **S6 — Semantic retrieval.** Larger; unlocks attachments and long-term memory.
